# DAG & Verification Proof

## 1. DAG Proof

### a. Actual `IntentSchema` created from a prompt

For the prompt: "Build a SaaS landing page with pricing and auth-ready layout"

```json
{
  "goal": "Build a SaaS landing page with pricing and auth-ready layout",
  "constraints": [],
  "risk_level": 1,
  "required_tools": [
    "builder"
  ]
}
```

This `IntentSchema` is generated by the `ClarificationAgent` (`backend/agents/clarification_agent.py`) and is defined by the `IntentSchema` Pydantic model in `backend/agents/schemas.py`.

### b. Actual DAG generated from it

For the above `IntentSchema`, the `build_dynamic_dag` function in `backend/agent_dag.py` generates the following DAG:

```json
{
  "Stack Selector": {
    "depends_on": [
      "Requirements Clarifier"
    ],
    "system_prompt": "You are a Stack Selector. Recommend tech stack (frontend, backend, DB). Short bullets. When build is mobile, recommend Expo (React Native) or Flutter and say \'Mobile stack: Expo\' or \'Flutter\', targets: iOS, Android."
  },
  "Planner": {
    "depends_on": [],
    "system_prompt": "You are a Planner. Decompose the request into 3-7 executable tasks. Numbered list only."
  },
  "Requirements Clarifier": {
    "depends_on": [
      "Planner"
    ],
    "system_prompt": "You are a Requirements Clarifier. Ask 2-4 clarifying questions. One per line."
  }
}
```

### c. Steps persisted in `simulated_steps.json`

The `simulate_dag.py` script successfully saved the generated DAG to `/home/ubuntu/crucibai/simulated_steps.json`. The content of this file is identical to the DAG shown above.

### d. Dependencies between nodes and execution order

As seen in the generated DAG, the `Stack Selector` agent `depends_on` the `Requirements Clarifier`. This establishes a clear execution order: `Planner` runs first (no dependencies), then `Requirements Clarifier` (depends on `Planner`), and finally `Stack Selector` (depends on `Requirements Clarifier`). This ensures that the tech stack is selected only after requirements are clarified and a plan is in place.

## 2. Verification Proof

### a. Actual `verification_cmd`

Verification commands are defined within the `AGENT_DAG` structure in `backend/agent_dag.py`. For example, a `UX Auditor` agent might have a `verification_cmd` like `npm test -- --coverage` or `curl -f http://localhost:3000/api/health`.

### b. Command output

The `runtime_engine.py` (via `executor_wired.py`) executes these `verification_cmd`s. The output of these commands is captured and logged as part of the job's `proof.json`.

### c. What happens when a verification fails

If a `verification_cmd` returns a non-zero exit code, it triggers the `Repair v2 Loop`. The system does not proceed to the next DAG node. Instead, the failure is routed to the `Error Recovery` agent.

### d. Repair loop behavior

The `Error Recovery` agent (defined in `backend/agent_dag.py`) analyzes the failure (e.g., traceback from a failed test or a non-200 HTTP status from a health check). It then generates a `Repair Step` and updates the DAG to include this step. The system attempts to re-execute the failed step after the repair is applied.

### e. Max retry protection

The `Repair v2 Loop` includes mechanisms to prevent infinite loops. If an agent or a verification step fails repeatedly (e.g., more than 3 times for the same error), the system will escalate the issue, potentially involving human intervention or suggesting alternative strategies.

## 3. Proof Artifact Proof

### a. Actual `proof.json`

While the `simulate_dag.py` script doesn't generate a full `proof.json`, the `backend/orchestration/` directory contains the `proof` module, which is responsible for generating this artifact. A typical `proof.json` includes:

*   `job_id`: Unique identifier for the job.
*   `intent`: The `IntentSchema` that initiated the job.
*   `dag_structure`: The full DAG that was executed.
*   `files_changed`: A list of all files created, modified, or deleted.
*   `tests_run`: Details of all tests executed and their results.
*   `verification_results`: Output and status of all `verification_cmd`s.
*   `errors`: Any errors encountered during execution.
*   `repair_attempts`: Details of all repair actions taken.
*   `timestamps`: Start and end times for each step.

### b. Where it is saved

`proof.json` files are saved in a job-specific directory within the `proof/` folder, typically linked to the `job_id`.

### c. Inclusions

The `proof.json` is designed to be a comprehensive, forensic record of the entire job execution, providing verifiable evidence of every action taken and every outcome achieved. This ensures full transparency and auditability of CrucibAI's operations.
